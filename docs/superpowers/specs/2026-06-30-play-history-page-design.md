# 历史播放记录页面设计

## 背景

用户希望在现有前端中新增一个历史播放记录页面，展示最近 100 首播放过的歌曲，每页 20 首，共 5 页。当前项目已有 `plays` 表和读取近期播放记录的基础能力，但播放路径需要补齐写入逻辑，历史页也需要独立 API，避免复用 `/api/now` 造成语义混杂。

目标是：后端保留完整原始播放事件，供后续统计分析使用；前端历史页展示最近 100 首唯一歌曲，同一首歌只显示最近一次播放。

## 范围

### 包含

- 新增专用历史接口 `GET /api/history?page=1&pageSize=20`。
- 补齐实际播放成功后的 `plays` 表写入。
- 新增前端 `HISTORY` tab 和历史面板。
- 历史页展示最近 100 首唯一歌曲，每页 20 首。
- 历史页支持空态、加载失败重试、分页和重新播放。
- 增加后端与前端测试覆盖。

### 不包含

- 不新增独立 URL 路由。
- 不修改统计分析的数据口径。
- 不在数据库层去重播放事件。
- 不增加搜索、筛选、删除历史等额外功能。

## 后端设计

### 数据模型

继续使用现有 `plays` 表：

- `song_id`
- `song_name`
- `artist`
- `played_at`

播放时始终追加原始事件，不做去重。这样重复播放同一首歌仍会被统计分析看到。

### 历史查询

新增一个专用于历史页的查询函数，语义是“最近播放过的唯一歌曲”：

1. 按 `song_id` 分组。
2. 每组保留最近一次播放记录。
3. 按最近 `played_at` 倒序排序。
4. 最多取最近 100 首唯一歌曲。
5. 对这个结果按 `page` 和 `pageSize` 分页。

历史页只展示有 `song_id` 的播放记录；没有 `song_id` 的异常记录仍可保留在原始表中，但不参与唯一歌曲列表。

接口响应：

```json
{
  "items": [
    {
      "song_id": "123",
      "song_name": "Song",
      "artist": "Artist",
      "played_at": "2026-06-30 12:00:00"
    }
  ],
  "page": 1,
  "pageSize": 20,
  "total": 100,
  "totalPages": 5
}
```

`pageSize` 默认 20，最大 20；展示总量最大 100，所以 `totalPages` 最大 5。

### 播放写入

在服务端拿到实际可播放曲目后写入 `plays`。需要覆盖聊天播放、搜索播放、FM/智能推荐、按 ID 播放、相似歌曲等最终返回曲目的路径。

写入点应尽量靠近“已经确定要返回给前端播放的曲目”的位置，避免记录失败搜索或仅存在于用户意图中的曲目。

## 前端设计

### 导航结构

沿用现有单页 tab/panel 模式：

- 在 `frontend/index.html` 增加 `HISTORY` tab。
- 增加 `history-panel` 容器。
- 在 `frontend/js/dom.js` 增加 DOM 引用。
- 新增 `frontend/js/history-panel.js`，由 `frontend/js/main.js` 在切换到 history tab 时调用。

不引入新框架、不新增客户端路由。

### 面板行为

历史面板进入时加载第 1 页。

每条记录展示：

- 歌曲名
- 歌手
- 最近播放时间
- 播放按钮

播放按钮调用现有 `/api/play/by-id`。播放成功后可以重新拉取当前页，使该歌曲因最近播放时间更新而回到更前位置，但仍只显示一次。

### 状态

- 加载中：显示简短 loading 状态。
- 空数据：显示“暂无播放记录”。
- 失败：显示错误提示和重试按钮。
- 分页：显示 1 到 `totalPages` 的页码按钮，当前页高亮。

样式复用现有 `.panel`、`.panel-empty`、`.track-item`、`.track-item-name`、`.track-item-artist`、`.track-action`。只为播放时间和分页补充少量必要样式。

## 测试计划

### 后端

- `GET /api/history` 默认返回第 1 页、每页 20 条。
- 当原始播放事件超过 100 条时，历史页总量最多 100 首唯一歌曲。
- 同一 `song_id` 多次播放时，接口只返回最近一次。
- 分页参数能返回正确页码和 `totalPages`。
- 实际播放路径成功返回曲目时会写入 `plays`。

### 前端

- `HISTORY` tab 和 panel 存在。
- 历史面板能渲染列表、页码和播放时间。
- 空数据时显示空态。
- 请求失败时显示错误和重试按钮。
- 播放按钮调用 `/api/play/by-id`。

### 手动验证

- 运行测试。
- 在非 3000 端口启动项目。
- 打开前端，触发至少一次播放。
- 进入 `HISTORY` tab，确认歌曲出现。
- 重复播放同一首歌，确认数据库可保留多条事件，前端只显示最近一次。
- 翻页检查每页最多 20 首、最多 5 页。

## 成功标准

- 历史页能稳定显示最近 100 首唯一歌曲。
- 后端 `plays` 表保留原始播放事件，不影响统计分析。
- 页面分页、空态、错态和重新播放行为可用。
- 新增或修改的测试通过。
- UI 在实际浏览器中完成基本验证。
